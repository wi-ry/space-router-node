# v0.2.0 Multi-Wallet Registration — Developer Guide

This document describes the changes introduced to support v0.2.0
registration while maintaining backward compatibility with v0.1.2.
Read this before working on registration, wallet management, or
Coordination API integration.

## What changed and why

v0.2.0 separates the single `wallet_address` into three role-specific
wallets:

| Wallet | Purpose | Has private key on node? |
|--------|---------|--------------------------|
| **Identity** | Node's persistent identity; signs all API requests | Yes — stored at `IDENTITY_KEY_PATH` |
| **Staking** | Locked in the on-chain staking contract; earns rewards | No — address only |
| **Collection** | Accumulates traffic fees | No — address only |

Any of the three can share the same address ("wallet collapsing").
When `STAKING_ADDRESS` or `COLLECTION_ADDRESS` is not configured,
it defaults to `WALLET_ADDRESS`.

## Architecture

### Registration dispatcher (`app/registration.py`)

Registration is now a strategy dispatch based on `SR_REGISTRATION_MODE`:

```
register_node()          ← public API (unchanged signature + node_address)
  ├─ mode="v1"  → _register_v1()    v0.1.2 payload
  ├─ mode="v2"  → _register_v2()    v0.2.0 payload
  └─ mode="auto"→ try v2, fallback to v1 on HTTP 400/422
```

Both flows POST to the same endpoint (`/nodes/register`) but with
different payload schemas. After a successful registration, both flows
call `request_probe()` to trigger server-side health verification.

A module-level `_active_mode` variable tracks which mode succeeded so
downstream code (e.g., future deregistration changes) can match the
protocol that was used.

### Payload differences

**v0.1.2 (`_register_v1`)**
```json
{
  "wallet_address": "0x...",
  "endpoint_url": "https://...",
  "identity_signature": "EIP-191 of space-router:register:{wallet_address}:{ts}",
  "timestamp": 1234567890,
  "label": "optional"
}
```

**v0.2.0 (`_register_v2`)**
```json
{
  "identity_address": "0x...",
  "staking_address": "0x...",
  "collection_address": "0x...",
  "staking_vouching_signature": "EIP-191 of space-router:vouch:{staking}:{collection}:{timestamp}",
  "identity_signature": "EIP-191 of space-router:register:{identity_address}:{ts}",
  "endpoint_url": "https://...",
  "timestamp": 1234567890,
  "label": "optional"
}
```

Key differences:
- v1 signs over `wallet_address`; v2 signs over `identity_address`
  (the address derived from the node's identity keypair).
- v2 adds a `vouching_signature` — a timestamp-free EIP-191 signature
  over `space-router:vouch:{staking_address}:{collection_address}:{timestamp}`.
  This binds the identity to both the staking and collection wallets.
- v2 never sends `wallet_address`; v1 never sends `identity_address`,
  `staking_address`, `collection_address`, or `vouching_signature`.

### Unchanged components

These are identical across both protocol versions:

- **Deregistration** (`PATCH /nodes/{id}/status`) — same signed payload
- **Health probe request** (`POST /nodes/{id}/request-probe`) — same signed payload
- **Challenge probe response** (`X-SpaceRouter-Address` header) — unchanged
- **Public IP detection** — unchanged
- **TLS/mTLS** — unchanged

## Files modified

| File | What changed |
|------|-------------|
| `app/config.py` | Added `STAKING_ADDRESS`, `COLLECTION_ADDRESS`, `REGISTRATION_MODE` settings with a `@field_validator` |
| `app/identity.py` | Added `sign_vouch(private_key, staking_address, collection_address)` |

| `gui/config_store.py` | Added `SR_REGISTRATION_MODE` to `_DEFAULTS` |
| `tests/test_registration.py` | Added 23 backward-compatibility tests across 7 new test classes |
| `docs/api-contracts.md` | Full API reference for both v0.1.2 and v0.2.0 contracts |

## New configuration

| Env var | Default | Description |
|---------|---------|-------------|
| `SR_REGISTRATION_MODE` | `v1` | `v1` = v0.1.2 payload, `v2` = v0.2.0 payload, `auto` = try v2 then fall back to v1 |
| `SR_STAKING_ADDRESS` | `""` (= `WALLET_ADDRESS`) | Staking wallet address for v0.2.0 |
| `SR_COLLECTION_ADDRESS` | `""` (= `WALLET_ADDRESS`) | Collection wallet address for v0.2.0 |

Existing env vars (`SR_WALLET_ADDRESS`, `SR_IDENTITY_KEY_PATH`, etc.)
are unchanged and still required.

## How to build on top of this

### Adding a new registration field

1. Decide which version(s) need it.
2. Add the field to the payload in `_register_v1` and/or `_register_v2`.
3. Add a test in the corresponding `TestRegisterNode` / `TestRegisterNodeV2` class.
4. If it's a v2-only field, add an assertion in `TestV1PayloadIsolation`
   confirming v1 payloads exclude it.

### Switching the default mode to v2

When the Coordination API is fully migrated to v0.2.0:

1. Change the default in `app/config.py`: `REGISTRATION_MODE: str = "v2"`
2. Change the default in `gui/config_store.py`: `"SR_REGISTRATION_MODE": "v2"`
3. Existing `v1` tests remain — they pin `REGISTRATION_MODE="v1"` in the fixture.

### Adding a new registration mode

1. Add the mode string to the validator in `app/config.py`.
2. Add a new `_register_<mode>()` function in `app/registration.py`.
3. Add a dispatch branch in `register_node()`.
4. Add a test fixture and test class.
5. Update `TestRegistrationModeConfig.test_valid_modes_accepted`.

### Making deregistration version-aware

Currently deregistration uses the same signed payload for both versions.
If v0.2.0 needs a different deregister format:

1. Add `_deregister_v2()` alongside the existing `deregister_node()`.
2. Use `_active_mode` to dispatch — it's already set by `register_node()`.
3. Add tests in `TestDeregisterPayloadConsistency` to verify both paths.

### Working with the auto-mode fallback

Auto mode tries `_register_v2` first. It falls back to `_register_v1`
only on HTTP 400 or 422 (the server doesn't understand v0.2.0 fields).
All other errors (403, 409, 500, connection errors) propagate immediately
and are retried by the backoff loop in `main.py`.

If you need to change which status codes trigger fallback, update the
check in `register_node()`:
```python
if exc.response.status_code in (400, 422):
```
and add a corresponding test in `TestAutoModePayloads`.

## Test organization

```
TestRegisterNode             — v1 registration (legacy, pinned to REGISTRATION_MODE="v1")
TestRegisterNodeV2           — v2 registration (multi-wallet)
TestAutoModeRegistration     — auto mode: v2 success, fallback, non-fallback errors
TestV1PayloadIsolation       — v1 payloads never leak v2 fields
TestSignatureMessageFormats  — v1 signs wallet_address, v2 signs identity_address
TestAutoModePayloads         — payload format inspection, 403/409 non-fallback
TestActiveModeTracking       — _active_mode set correctly per mode
TestRegistrationModeConfig   — config validation and defaults
TestV2EdgeCases              — UPnP, empty label
TestDeregisterPayloadConsistency — deregister format unchanged across modes
TestSignVouch                — vouching signature unit tests
```

All existing tests are pinned to `REGISTRATION_MODE="v1"` via the
`reg_settings` fixture. They will not break when the default mode
changes in the future.
